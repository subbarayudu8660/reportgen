## Commands

- `npm install` ŌĆö install dependencies
- `npm run dev` / `npm start` ŌĆö start the Express server (both run `node server.js`; no build step, no watcher)
- Server refuses to boot if `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, or `GOOGLE_REDIRECT_URI` are missing from `.env` (checked in `server.js`). `GA4_PROPERTY_ID`/`GOOGLE_SHEET_ID` are no longer required at boot ŌĆö they're only used as one-time fallback defaults to seed the first client (see Client management below). If `SESSION_SECRET` is missing, `server.js` generates a random one and appends it to `.env` on first boot (see OAuth / token handling below) rather than refusing to start.
- There is no test suite, linter, or build tooling configured

## Architecture

ReportGen is a Node/Express app that authenticates against Google via OAuth 2.0, pulls GA4 report data for one or more clients, and generates a downloadable `.pptx` comparing two user-chosen reporting periods using `pptxgenjs`. Frontend is plain HTML/JS served statically from `public/`.

### Request flow

1. `public/app.js` polls `GET /auth/status` to decide whether to show the "Sign in" step or the "Choose Reporting Periods" step, and fetches `GET /api/clients` to populate the client selector independently of auth state.
2. `GET /auth/google` (routes/auth.js) redirects to Google's consent screen; `GET /auth/callback` exchanges the code for tokens via `services/googleAuth.js` and persists them to `tokens.json` at the repo root (gitignored, mode 0600).
3. `POST /api/generate-report` (routes/report.js) takes `{ clientId, currentMonth: "YYYY-MM", comparisonMonth: "YYYY-MM", paidMedia }` ŌĆö both periods are picked by the user and are not required to be adjacent months; `paidMedia` is optional manual-entry ad data (see Paid Media slide below). It looks up the client via `services/clients.js#getClientById`, calls `services/ga4.js#getFullReportData(currentMonth, comparisonMonth, client.ga4PropertyId)` and (if `client.sheetId` is set) `services/sheets.js#getSeoOverview(currentMonth, comparisonMonth, client.sheetId)` in parallel, and ŌĆö if `client.metaAdAccountId` is set ŌĆö separately fetches `services/meta.js#getMetaAdsData(client.metaAdAccountId, currentMonth)` (see Meta Ads data layer below), and ŌĆö if `client.googleAdsCustomerId` is set ŌĆö separately fetches `services/googleAds.js#getGoogleAdsData(client.googleAdsCustomerId, currentMonth, email)` (see Google Ads data layer below). It merges the SEO result onto the GA4 data as `data.seo`, and builds `data.paidMedia = { googleAds: <API data if a Google Ads customer ID is configured, else manual form data>, metaAds: <API data if account configured, else manual form data> }` (see Paid Media slide below), then pipes the combined object into `services/pptx.js#generateReportPptx`, and streams the resulting buffer back as a file download. There is no `GET /api/months` endpoint ŌĆö the month/year selectors are built entirely client-side in `public/app.js`. Neither GA4 nor Sheets service reads `GA4_PROPERTY_ID`/`GOOGLE_SHEET_ID` from `.env` directly anymore ŌĆö every call is parameterized by the active client's IDs, and there is no hardcoded "previous month" derivation anywhere in the backend.

### Client management (`services/clients.js`, `routes/clients.js`)

- Clients are persisted as a flat array in `clients.json` at the repo root (gitignored, like `tokens.json`) ŌĆö `{ id, name, ga4PropertyId, sheetId, metaAdAccountId, googleAdsCustomerId, websiteUrl, createdAt }`. `id` is a `crypto.randomUUID()`.
- `getClients()` calls `ensureSeeded()`, which creates a single "Default Client" from `.env`'s `GA4_PROPERTY_ID`/`GOOGLE_SHEET_ID`/`META_AD_ACCOUNT_ID` **only** the very first time the app runs (`clients.json` doesn't exist yet on disk) ŌĆö `googleAdsCustomerId` is not seeded from `.env` (there's no single-client env fallback for it) and defaults to `''`. Once the file exists ŌĆö even if the user later deletes every client down to an empty array ŌĆö it is never reseeded, so `.env` values are a one-time bootstrap default, not a live fallback.
- `GET /api/clients` lists clients; `POST /api/clients` validates `name`+`ga4PropertyId` are present (`sheetId`, `metaAdAccountId`, and `googleAdsCustomerId` are all optional ŌĆö a client with no sheet ID skips the Sheets call entirely in `routes/report.js` and renders the SEO slide as "Not configured", same as any other empty-data case; a client with no `metaAdAccountId` skips the Meta Ads API call and falls back to the manual Meta Ads form fields, or "Not configured" if those are empty too; a client with no `googleAdsCustomerId` skips the Google Ads API call and falls back to the manual Google Ads form fields the same way) and appends a new client; `DELETE /api/clients/:id` removes one (404 if not found). There is no update/edit endpoint ŌĆö editing a client means deleting and re-adding it.
- `normalizeMetaAdAccountId()` in `services/clients.js` prepends `act_` to whatever the user types if it isn't already present, since the Meta Graph API requires that prefix on ad account IDs but users commonly copy just the bare numeric ID from Ads Manager. Applied both in `ensureSeeded()` (from `.env`) and `addClient()` (from the form).
- `normalizeGoogleAdsCustomerId()` in `services/clients.js` strips dashes from whatever the user types (e.g. `123-456-7890` ŌåÆ `1234567890`), since the Google Ads API requires the bare 10-digit customer ID but Ads Manager displays it dashed. Applied in `addClient()` only ŌĆö there's no `.env`-seeded default for this field.
- `websiteUrl` (plain text, e.g. `fashionrerun.com`) is an optional per-client field, not normalized in any way (no protocol stripping/prefixing) ŌĆö it must match the `website` column value stored in Supabase's `scans` table exactly. Used only by `services/supabase.js` (see Supabase keyword data source below); has no `.env`-seeded default and no effect on any other slide.

### Period selection & report warnings (frontend)

- `public/index.html`/`public/app.js` show **two** independent Month+Year picker pairs, "Current Period" and "Comparison Period", instead of a single month dropdown. `populatePeriodSelectors()` builds both client-side: Current Period defaults to the most recently completed month (this month minus one, rolling back a year in January), and Comparison Period defaults to one month before that. Either period can be changed to any month/year in the current year or prior 2 years ŌĆö they don't need to be adjacent. The Generate button starts `disabled` and `updateGenerateButtonState()` re-enables it once both period pairs have a value **and** a client is selected.
- The download filename is `WebrocketAI-Report-<CurrentMonth>-<CurrentYear>-vs-<ComparisonMonth>-<ComparisonYear>.pptx`, built from `monthLabel()` (services/pptx.js) server-side for `Content-Disposition` (routes/report.js) and mirrored client-side for the `<a download>` fallback name.
- A collapsible `<details id="paid-media-section">` sits between the period pickers and the Generate button, labeled "Optional ŌĆö fill in if the client runs paid ads". It has two side-by-side subsections, Google Ads and Meta Ads, with plain number/text `<input>`s (no client-side required validation ŌĆö every field is optional at the form level). `collectPaidMedia()` in `public/app.js` reads all of them on submit via `numOrNull`/`strOrNull` helpers (empty string ŌåÆ `null`, otherwise `Number(...)` or the trimmed string) and sends the result as `paidMedia: { googleAds: {...}, metaAds: {...} }` in the `POST /api/generate-report` body regardless of whether anything was filled in. **The manual Google Ads and Meta Ads fields here are only used as a fallback** ŌĆö if the active client has a `googleAdsCustomerId`/`metaAdAccountId` configured (respectively), `routes/report.js` discards the matching manual object entirely and substitutes live API data instead (see Google Ads / Meta Ads data layers below). Whether the resulting slide appears is decided server-side in `services/pptx.js` (see PPTX generation below), not by the frontend.
- `routes/report.js` computes a `warnings` array (plain user-facing sentences) after generating the pptx ŌĆö one per: no GA4 data at all for the current period (`gaData.hasAnyGa4Data` false), no keyword ranking data (`seoData.keywordRankings.hasData` false ŌĆö text is `seoData.keywordRankings.error` when the sheet's tab-name resolution failed, else the generic "no date column" message), no off-page data (`seoData.offPage.hasData` false, same `error`-vs-generic preference), no comparison-period GA4 data (only added when the current period *does* have data, to avoid a redundant message alongside "no GA4 data at all"), a Meta Ads fetch failure (`"Meta Ads data could not be fetched: <reason>"`, only when `client.metaAdAccountId` is set and the fetch threw), and a Google Ads fetch failure (only when `client.googleAdsCustomerId` is set and the fetch threw ŌĆö `GOOGLE_ADS_TOKEN_REQUIRED` and `GOOGLE_ADS_PERMISSION_DENIED` get their own specific wording, see Google Ads data layer above; any other error falls back to `"Google Ads data could not be fetched: <message>"`). It's sent as a `X-Report-Warnings` response header (JSON, `encodeURIComponent`-escaped, since header values must be ASCII) alongside the binary pptx ŌĆö there's no separate JSON response for this, since the endpoint always returns the file on success. `public/app.js` reads and decodes this header, and renders each warning in the `#warnings` box (amber, non-blocking) alongside the download.

### OAuth / token handling (`services/googleAuth.js`)

- Tokens live in a single local JSON file (`tokens.json`), not a database, but the file holds **multiple users' tokens** keyed by Google account email: `{ "user@gmail.com": { access_token, refresh_token, expiry_date, scope, ... } }`. This app is still local-only by design (no external DB), but multiple people can sign in with different Google accounts against the same running server without clobbering each other's credentials.
- Every function that reads or writes tokens takes an `email` argument: `loadTokens(email)`, `saveTokens(email, tokens)`, `isAuthenticated(email)`, `hasSheetsScope(email)`, `getAuthorizedClient(email)`. There is no more "the" token ŌĆö always pass the email of the user whose credentials you want.
- The signed-in user's email is identified via the Google userinfo endpoint (`GET https://www.googleapis.com/oauth2/v2/userinfo`, called with the freshly-exchanged access token) inside `handleCallback(code)`, which now returns `{ tokens, email }` instead of just `tokens`. `SCOPES` includes `openid` and `.../auth/userinfo.email` (in addition to `analytics.readonly` and `spreadsheets.readonly`) so that call succeeds.
- Google only returns a `refresh_token` on the *first* consent grant for a given account; `handleCallback` and the `client.on('tokens', ...)` listener in `getAuthorizedClient` both special-case this by merging new tokens onto that email's existing stored `refresh_token` rather than overwriting it.
- `getAuthorizedClient(email)` throws `NOT_AUTHENTICATED` (no email, or no stored refresh token for that email) or `AUTH_EXPIRED` (refresh failed) ŌĆö callers key off `err.code` to return the right HTTP status.
- `hasSheetsScope(email)` checks that email's stored token `scope` string for the Sheets scope. Tokens saved before the Sheets scope was added won't have it; `GET /auth/status` reports `needsReauth: true` in that case (authenticated but missing scope), and `public/app.js` shows the sign-in step with a prompt instead of the generate step. `services/sheets.js` also guards each call with its own `ensureSheetsScope(email)`, throwing `SHEETS_SCOPE_MISSING` (mapped to HTTP 401 in `routes/report.js`) as a second line of defense.
- **Which user is "current" is a session concern, not a token-store concern.** `server.js` wires up `express-session` (cookie name `reportgen.sid`, backed by `SESSION_SECRET` from `.env`) before the static/route middleware. On first boot, if `SESSION_SECRET` is absent, `server.js` generates one with `crypto.randomBytes(32)` and appends it to `.env` ŌĆö it does not block startup or require the user to set it manually. `routes/auth.js#GET /callback` stores the resolved email as `req.session.email` after a successful OAuth exchange; every downstream route that needs Google API access reads `req.session.email` and threads it down as the `email` argument (`routes/report.js` ŌåÆ `services/ga4.js#getFullReportData(..., email)` / `services/sheets.js#getSeoOverview(..., email)` ŌåÆ every individual GA4/Sheets fetch function ŌåÆ `getAuthorizedClient(email)`). `GET /auth/status` reads `req.session.email` and returns `{ authenticated, needsReauth, email }` for the current session only ŌĆö two different browser sessions (e.g. two users, or one user in a normal tab vs. incognito) see independent auth state even though they share the same `tokens.json`.
- `GET /auth/logout` calls `req.session.destroy()`, clears the `reportgen.sid` cookie, and redirects to `/` ŌĆö this only ends the *current session*, it does not delete anything from `tokens.json`, so other signed-in users (or the same user in another browser/session) are unaffected. It's a plain `<a href="/auth/logout">` (not a fetch call) so the redirect does a full page reload, which re-runs `checkAuthStatus()` and lands back on the sign-in screen. There is no endpoint to remove a user's stored tokens entirely; the only way to fully revoke is to edit `tokens.json` or revoke access from the Google Account permissions page.
- `public/index.html` has a `#user-bar` fixed to the top-right corner of the page (email + "Sign Out" link, styled `.btn-outline` ŌĆö white/grey/dark, deliberately unobtrusive), outside `.container` so it stays in place independent of card layout/scroll. `checkAuthStatus()` in `public/app.js` toggles it on `authenticated` from `GET /auth/status` (shown even when `needsReauth` is true, since signing out is a valid way to recover from that state) and fills in the email from the same response.
- Migration note: `tokens.json` files written before this change are a flat single-token object (no email keys), which `loadAllTokens()` silently treats as "no known users" ŌĆö anyone who was signed in under the old scheme must sign in again once after this change ships.

### GA4 data layer (`services/ga4.js`)

- Calls the GA4 Data API REST endpoint directly (`analyticsdata.googleapis.com/v1beta/properties/{id}:runReport`) via `fetch` with a bearer token from the OAuth client, rather than using the `@google-analytics/data` gRPC SDK ŌĆö simpler to reason about with user-OAuth (vs. service-account) credentials. `runReport(body, propertyId)` takes the property ID as an explicit argument (the active client's `ga4PropertyId`) rather than reading `GA4_PROPERTY_ID` from `.env`.
- `runReport()` throws `GA4_API_ERROR` on any non-2xx response, with `.status` (HTTP status) and `.apiErrorStatus` (the GA4 API's own error status string parsed from the JSON body, e.g. `PERMISSION_DENIED`, `NOT_FOUND`, `INVALID_ARGUMENT`, `RESOURCE_EXHAUSTED`, `UNAUTHENTICATED` ŌĆö `null` if the body wasn't JSON). `mapGa4Error(err)` maps that error to a specific, user-actionable `{ httpStatus, message }` pair (403 permission/property-access, 404 property not found, 400 invalid property ID format, 429 quota exceeded, 401 expired session, else a generic "unexpected error (code)" at 502); `routes/report.js` logs the raw `status`/`apiErrorStatus`/`message` server-side and returns `mapGa4Error`'s result as the JSON error response. Any new GA4-error-handling case should extend `mapGa4Error`, not add ad hoc status checks in `routes/report.js`.
- All four report sections are fetched in parallel via `Promise.all` inside `getFullReportData(currentMonth, comparisonMonth, propertyId)`: organic search metrics, ecommerce funnel, top landing pages, and channel traffic overview ŌĆö each fetched for both the current and comparison month (both user-supplied; there is no assumption that the comparison month is adjacent to the current one) so percent-change can be computed.
- `getTopLandingPages` does a second query (`getSessionsByPagePath`) scoped to the current month's top-10 page paths to get comparison-month sessions for the *same* pages (not the comparison month's own top 10) ŌĆö pages with no comparison-month traffic get `prevSessions: null` / `change: null`, rendered as "N/A".
- `getSiteTotals(monthStr, propertyId, email)` fetches unfiltered, account-wide `screenPageViews`/`activeUsers` for the **current** period only (same `runReport()`/`sumMetric()` pattern as `getMonthSessionTotal`), and derives `pagesPerUser = screenPageViews / activeUsers` rounded to 2 decimals (`null` if `activeUsers` is 0). `getFullReportData` fetches this alongside the other sections and merges it onto `landingPages` (`data.landingPages = { pages, hasData, screenPageViews, activeUsers, pagesPerUser }`) ŌĆö these three fields feed the Top Landing Pages slide's hero stat cards in `pptx.js`, not a separate top-level field.
- Ecommerce `hasData` is computed by summing all funnel metrics + revenue and checking `> 0`, not by row count ŌĆö GA4 can return rows for zero-count events, so row presence alone is not a reliable "no ecommerce" signal.
- `getFullReportData` also fetches an unfiltered total-sessions count for both months (`getMonthSessionTotal`) to derive `hasAnyGa4Data`/`comparisonMonthHasGa4Data` ŌĆö these are channel/event-agnostic, unlike the per-section `hasData` flags, so they answer "did GA4 track *anything* this month" (e.g. before the property existed) rather than "did this specific section have rows". When `comparisonMonthHasGa4Data` is false, `organicSearch.changes`, `ecommerce.changes`, and each `trafficOverview` row's `change` are forced to `null` instead of being computed against a false "0" baseline ŌĆö a real zero-traffic comparison month and a not-yet-tracked comparison month must not both render as "+100%"/"0%". The returned object's top-level fields are `currentMonth`/`comparisonMonth` (not `month`/`previousMonth`).

### Meta Ads data layer (`services/meta.js`)

- Calls the Meta Marketing/Graph API directly via `fetch` (native Node 18 `fetch`, same pattern as `ga4.js`/`sheets.js`) using the single app-level `META_ACCESS_TOKEN` from `.env` ŌĆö there is no per-client OAuth for Meta, only the ad account ID is per-client (`client.metaAdAccountId`). API version is pinned at `v19.0` (`META_API_VERSION` const).
- `getMetaAdsData(adAccountId, monthStr)` makes two parallel calls: `GET /{ad_account_id}/insights` (`time_range: { since, until }` derived from the `"YYYY-MM"` month via `monthDateRange()`, `fields=campaign_name,impressions,clicks,spend,reach,actions`, `level=account`) for account-level totals, and a second `GET /{ad_account_id}/insights` at `level=campaign` (`fields=campaign_id,spend`) to compute `totalCampaigns`. Returns `{ totalCampaigns, totalSpend, impressions, clicks, reach, formLeads, messageConversations }` ŌĆö `totalCampaigns` counts the distinct `campaign_id`s from the campaign-level insights call whose `spend > 0` during the reporting date range, **not** campaigns whose *current* `status` is `ACTIVE` ŌĆö a campaign that spent money during the period but was since paused/archived still counts, and a campaign that's enabled but spent nothing does not; `totalSpend` is formatted as a `$`-prefixed string via `fmtSpend()` to match the manual-entry currency-string convention used elsewhere in the Paid Media slide; `formLeads`/`messageConversations` are pulled out of the `actions` array via `sumAction()` matching `action_type === 'lead'` and `action_type === 'onsite_conversion.messaging_conversation_started_7d'` respectively (0 if the action type isn't present, not `null`, since "no leads" and "not tracked" aren't distinguished by this API).
- Throws with a `.code` the caller switches on ŌĆö `META_NOT_CONFIGURED` (no `META_ACCESS_TOKEN` in `.env`), `META_NO_ACCOUNT` (falsy `adAccountId` argument), `META_TOKEN_EXPIRED` (Graph API error code `190` ŌĆö message is the exact user-facing string asked for: "generate a new token and update META_ACCESS_TOKEN"), or `META_API_ERROR` (any other non-2xx/error response, or a network-level fetch failure). None of these are thrown as generic `Error` ŌĆö always check `.code`, following the same pattern as `GA4_API_ERROR`/`SHEETS_API_ERROR`.
- `routes/report.js` calls this in its own `try/catch`, separate from the GA4/Sheets `Promise.all` ŌĆö a Meta failure never fails the whole report; it just leaves `data.paidMedia.metaAds` as `null` (so the slide's Meta Ads section renders "Not configured for this period") and adds one entry to the `warnings` array. Skipped entirely (no call made) when `client.metaAdAccountId` is falsy.

### Google Ads data layer (`services/googleAds.js`)

- Calls the Google Ads API REST endpoint directly via `fetch` (`https://googleads.googleapis.com/{GOOGLE_ADS_API_VERSION}/customers/{customerId}/googleAds:search`, `GOOGLE_ADS_API_VERSION` const currently `v24`), not the Node client library ŌĆö same fetch-based pattern as `ga4.js`/`sheets.js`/`meta.js`. Check https://developers.google.com/google-ads/api/docs/release-notes when bumping this constant.
- Unlike Meta (single app-level token), Google Ads reuses the **signed-in user's own OAuth token** ŌĆö `getGoogleAdsData(googleAdsCustomerId, monthStr, email)` calls `services/googleAuth.js#getAuthorizedClient(email)` for the bearer token, same as `ga4.js`. Every request additionally sends a `developer-token` header (`GOOGLE_ADS_DEVELOPER_TOKEN` from `.env`, app-level) and a `login-customer-id` header (`GOOGLE_ADS_LOGIN_CUSTOMER_ID` from `.env`, the MCC account ID) ŌĆö both required by the API regardless of whose OAuth token is used.
- Runs a single GAQL query via `googleAds:search`: `SELECT campaign.name, campaign.status, metrics.impressions, metrics.clicks, metrics.cost_micros, metrics.conversions FROM campaign WHERE segments.date BETWEEN '<start>' AND '<end>'`, date range from `monthDateRange()` (services/utils.js). Returns `{ totalCampaigns, totalSpend, impressions, clicks, conversions, cpc }` ŌĆö field names deliberately match the Meta/manual-entry paid-media shape (`impressions`/`clicks`/`conversions`, not `total*`) so `services/pptx.js` can render Google Ads and Meta Ads data uniformly. `totalCampaigns` counts campaigns with at least one impression in the period (not all campaigns returned by the query); `totalSpend` and `cpc` are `$`-prefixed currency strings (`cost_micros` summed and divided by 1,000,000 for spend; spend/clicks for cpc, `$0.00` if no clicks). `cpc` is the signal `services/pptx.js` uses to detect "this is live API data" vs. a manual form entry (the manual form has no CPC field).
- Throws with a `.code` the caller switches on ŌĆö `GOOGLE_ADS_NOT_CONFIGURED` (missing `GOOGLE_ADS_DEVELOPER_TOKEN` or `GOOGLE_ADS_LOGIN_CUSTOMER_ID` in `.env`), `GOOGLE_ADS_NO_ACCOUNT` (falsy `googleAdsCustomerId` argument), `GOOGLE_ADS_TOKEN_REQUIRED` (developer token still in test-account mode / pending Basic Access approval ŌĆö detected via `developerTokenError`/`authenticationError` in the API's error details), `GOOGLE_ADS_PERMISSION_DENIED` (`authorizationError` in error details, or a bare 403 ŌĆö the signed-in OAuth account lacks access to that customer ID), or `GOOGLE_ADS_API_ERROR` (any other non-2xx response or network failure). Same pattern as `META_*`/`GA4_API_ERROR` ŌĆö always check `.code`, never a generic `Error`.
- `routes/report.js` calls this in its own `try/catch`, separate from GA4/Sheets/Meta ŌĆö a Google Ads failure never fails the whole report. `GOOGLE_ADS_TOKEN_REQUIRED` and `GOOGLE_ADS_PERMISSION_DENIED` each produce a specific user-facing warning string (see Period selection & report warnings above); any other error produces a generic `"Google Ads data could not be fetched: <message>"` warning; `GOOGLE_ADS_NO_ACCOUNT` is skipped silently (shouldn't occur in practice since the call is gated on `client.googleAdsCustomerId` being truthy). Skipped entirely (no call made) when `client.googleAdsCustomerId` is falsy ŌĆö falls back to the manual Google Ads form fields from `paidMedia.googleAds`, same fallback pattern as Meta Ads.

### Supabase keyword data source (`services/supabase.js`)

- **Additive, not a replacement.** This is an alternative keyword-ranking data source that renders as its own extra slide ("Keyword Rankings ŌĆö Live Tracking"), fully separate from the existing Sheets-based SEO Performance Overview slide. It exists so the Supabase data can be visually verified before any decision is made to merge or replace the Sheets-based logic. Off-page submissions and Domain/Page Authority continue to come exclusively from Google Sheets (`services/sheets.js`) regardless of whether a client has `websiteUrl` set ŌĆö Supabase only ever supplies keyword ranking buckets.
- Uses the `@supabase/supabase-js` client, instantiated lazily (and cached) in `getSupabaseClient()` from `SUPABASE_URL`/`SUPABASE_SERVICE_KEY` in `.env` (service-role key ŌĆö server-side only, never exposed to the frontend). Throws `SUPABASE_NOT_CONFIGURED` if either is missing.
- `getKeywordRankings(websiteUrl, currentMonth, comparisonMonth)` first queries `scans` (`SELECT id, scan_date, country WHERE website = websiteUrl ORDER BY scan_date DESC`) ŌĆö throws `SUPABASE_NO_SCANS` if the site has never been scanned at all. For each of the two periods independently, `pickScanForMonth()` filters scans to that calendar month (UTC bounds, since `scan_date` is a `timestamptz`) and picks the one with the **latest `scan_date`**; if multiple scans share that exact latest timestamp (e.g. different countries scanned same day), it prefers `country = 'in'`, else takes the first. The selected scan's `id`/`scan_date`/`country` is logged via `console.log` for debugging. If a period has no scan in that month, its bucket set is `null` (not zeroed) and a warning is appended: `"No keyword scan found for <Month Year> for <site>. Available scan dates: [...]"` (the 5 most recent scan dates for that site, regardless of month).
- For each selected scan, `fetchBucketsForScan()` queries `ranking_results` (`SELECT keyword, rank WHERE scan_id = X`) and buckets every row via `classifyRank()`: parses `rank` as an integer when possible (1-10 ŌåÆ `top10`, 11-30 ŌåÆ `top11to30`, 31-50 ŌåÆ `top31to50`, 51-100 ŌåÆ `top51to100`); the literal string `'Not Found'`, any non-numeric value, or a missing row all classify as `pending`. Note these bucket key names (`top11to30`, not `top11_30`) are Supabase-specific and intentionally distinct from the Sheets-based `KEYWORD_BUCKETS` keys in `services/pptx.js` ŌĆö the two data sources are not meant to share a shape yet.
- Both queries are wrapped in a 15-second timeout (`withTimeout()`) that throws `SUPABASE_TIMEOUT` ŌĆö Supabase JS has no built-in request timeout, so this guards against a hung connection blocking report generation indefinitely. Any other non-2xx/query error throws `SUPABASE_API_ERROR`. Same "always check `.code`" pattern as `META_*`/`GOOGLE_ADS_*`/`GA4_API_ERROR`.
- Returns `{ current, comparison, currentScan, comparisonScan, warnings }` ŌĆö `current`/`comparison` are each either a bucket-count object or `null`; `currentScan`/`comparisonScan` are `{ id, scanDate, country }` or `null`, used by the slide's caption; `warnings` is the list of human-readable "no scan this month" messages (0, 1, or 2 entries).
- `routes/report.js` calls this in its own isolated `try/catch`, separate from GA4/Sheets/Meta/Google Ads ŌĆö a Supabase failure never fails the whole report; it just omits the new slide (`data.supabaseKeywords` stays `null`) and adds one warning to `X-Report-Warnings` (`SUPABASE_NOT_CONFIGURED`, `SUPABASE_NO_SCANS`, `SUPABASE_TIMEOUT`, or a generic message for `SUPABASE_API_ERROR`). Skipped entirely (no call made) when the active client has no `websiteUrl` configured. On success, any per-period "no scan this month" warnings from `getKeywordRankings()` are also appended to `X-Report-Warnings`.

If exactly one of the two has data, the other renders as a "Not configured for this period" line instead of a table. Missing optional fields within a filled-in manual section (e.g. `conversions`, `gstAmount`) render as `ŌĆö`, not "Not configured" ŌĆö that label is reserved for a fully-empty section. ### Google Sheets data layer (`services/sheets.js`)

- Reads tabs of the spreadsheet at a given `sheetId` (the active client's `sheetId`, not read from `.env`) via the Sheets API v4 REST endpoint (`sheets.googleapis.com/v4/spreadsheets/{id}/values/{range}`) using the same bearer token pattern as `ga4.js`, not the `googleapis` SDK. `getSheetValues(range, sheetId)` throws if `sheetId` is falsy.
- **Tab names are resolved with a fast-path exact match, falling back to dynamic substring scanning.** `getSeoOverview` calls `getSheetTabTitles(sheetId, email)` (`GET .../spreadsheets/{id}?fields=sheets.properties.title`) to list the spreadsheet's actual tab titles, then `resolveTabs(tabTitles)` resolves each category via `resolveTab(exactVariants, dynamicPredicate, titles)`: first `exactMatchTab()` tries an exact (case/whitespace/apostrophe/backtick-insensitive) match against `TAB_VARIANTS.keywords` (`"Keyword Ranking Report"`, `"Keywords Ranking Report"`, `"Keyword Rankings"`), `TAB_VARIANTS.kpi` (`"KPI's"`, `"KPIs"`, `"KPI"`, `"Monthly KPIs"`), or `TAB_VARIANTS.offpage` (`"KPI_2"`, `"KPI 2"`, `"Off-page SEO Submission Tracker"`) ŌĆö this is just a shortcut for names we've actually seen, not the source of truth. If nothing matches exactly, it falls back to a `.includes()` scan of every tab title: a tab is the **keyword tab** if its name contains `"keyword"`; the **off-page tab** if it contains `"kpi_2"`, `"kpi 2"`, `"off-page"`, `"offpage"`, `"submission"`, or `"activity tracker"`; the **KPI traffic/authority tab** if it contains `"kpi"` and isn't already classified as off-page (so `"KPI's"`/`"KPI"`/`"KPIs"` land here, `"KPI_2"`/`"KPI 2"` do not ŌĆö off-page is resolved first specifically so it can never be swallowed by the more permissive KPI-traffic fallback). When more than one tab matches a category (e.g. `"Keywords"` and `"Keywords Old"`), `pickBestTab()` filters out names containing `old`/`archive`/`archived`/`backup`/`deprecated` and takes the first remaining match. `quoteSheetName()` doubles internal single quotes (`KPI's` ŌåÆ `'KPI''s'`) so apostrophe'd tab names build a valid A1 range literal. If a category matches no tab (neither fast path nor fallback), that section's fetch is skipped entirely and the section gets `hasData: false, error: "<Section> tab not found. Available tabs: [...]."` ŌĆö `routes/report.js` prefers this `error` string over its generic warning text when present. When a tab *is* resolved but simply has no date/month column for the requested period, `error` names the specific tab and lists the date columns that *were* found, instead of the generic "no data for this month" text.
- **Universal label matcher**: `findRowByLabel(rows, columnIndex, variants)` (`rows.findIndex` + `labelContainsAny`) is the one function used everywhere a "which row is X in" question needs answering ŌĆö keyword header detection, DA/PA rows. It never assumes a fixed row number, and the first match wins if a label appears more than once. `labelContainsAny`/`matchesVariant` normalize both sides with `normalizeLabel()` (trim, lowercase, curly-apostrophe/backtick ŌåÆ `'`) before comparing; **short variants (Ōēż3 chars, e.g. `"da"`/`"pa"`) require an exact match rather than substring-contains**, specifically so `"DA"` doesn't false-positive against `"Data Studio Link"` and `"PA"` doesn't false-positive against `"Paid Search"`. `isBlankCell()` treats a cell as empty if, after normalization, stripping all non-alphanumeric characters leaves nothing ŌĆö so a stray `'`, `` ` ``, or `--` left over from a copy-paste is treated as blank, not as a real label or an end-of-data marker.
- **Keyword rankings** (`'{tabName}'!A1:ZZ2000`): the header row is found dynamically via `findKeywordHeaderRowIndex()` ŌĆö any row anywhere in the tab whose column A contains `"keyword"` (not assumed to be row 10 or 11) ŌĆö so header position varies per client without code changes (verified against a header at row 12, a decorative title row + blank spacer rows before the real header, and labels at arbitrary/random positions). When more than one row matches (e.g. a title row like `"Keyword Ranking Report"` *and* the real header both contain "keyword"), the candidate with the most parseable date columns beside it wins, ties going to the lower (later) row, since a decorative title always sits above its header. Data rows run from immediately after the header until the first row with a blank column A (`isBlankCell()`). Date columns are found by `findDateColumns()` parsing every header cell from column B onward with `parseFlexibleDate()` (see below); `latestColForMonth()` picks the *most recent* matching date column when a sheet has multiple snapshots in the same month (e.g. both `6/15/2026` and `6/30/2026` in June ŌĆö `6/30` wins), and logs (`console.log`) which date column it picked for each period.
- **Each period's bucket set is `null`, not zeroed, when that specific period has no matching date column** ŌĆö e.g. requesting `2026-06` vs `2026-05` against a sheet whose only columns are `6/15/2026`/`6/30/2026`: `current` is a real bucket count (using `6/30`), `previous` is `null`, and `keywordRankings.comparisonError` is set to `"No keyword ranking data found for May 2026 ŌĆö nearest available dates in sheet: [6/15/2026, 6/30/2026]."` (`routes/report.js` surfaces this in `X-Report-Warnings` whenever the current period *does* have data, mirroring the existing GA4 "no comparison period data" pattern). `keywordRankings.error` is reserved for when the **current** period itself has no data (or the tab/header can't be found at all) ŌĆö `hasData` is `false` in that case. Per-bucket `changes[key]` is `undefined` (not `pctChange`'s own `null` "New" sentinel) whenever either period's bucket set is `null`, so `fmtPct()` in `pptx.js` renders `"N/A"` rather than a fabricated percentage or "New".
- Each keyword's ranking value is bucketed by `classifyRank`: Top 10 / 11-30 / 31-50 / 51-100 / Pending (100+), where blank, non-numeric, and any value containing `"100+"` are all treated as Pending.
- **Off-page/submissions tab** (`'{tabName}'!A1:ZZ500`): row 1 is dates per-day across columns (parsed with the same `parseFlexibleDate()`), rows 2+ are named activity types. All activity rows are summed across every date column that falls in the target month ŌĆö a total across activity types, not per-type.
- **`parseFlexibleDate(raw)`** (replaces the old `M/D/YYYY`-only `parseSheetDate`) tolerantly parses, in order: ISO `YYYY-MM-DD`; slash dates `M/D/YYYY` (assumed month-first, but falls back to day-first `D/M/YYYY` when the first part isn't a valid month); spelled-out month names/abbreviations (`"June 1, 2026"`, `"Jun 1 2026"`, `"1 June 2026"`); and bare Google Sheets/Excel serial numbers (days since 1899-12-30, only treated as a date when the number falls in a plausible calendar range ~1955ŌĆō2064, so ordinary numeric cells like rankings aren't misread as dates). Any cell that doesn't match any of these returns `null` and is silently skipped as "not a date column" ŌĆö this never throws. `parseMonthLabel(raw)` is the sibling used for the KPI tab's bare month headers (`"JAN"`, `"January"`, `"01"`, `"1"`, optionally with a trailing year) rather than full dates. `monthYearLabel(monthStr)` formats `"2026-06"` as `"June 2026"` for warning text.
- **KPI tab ŌĆö Domain/Page Authority only** (`getKpiAuthority()`). Organic/Direct/Referral/Social/Paid-Search traffic is deliberately **not** pulled from this tab ŌĆö GA4 (`services/ga4.js`) is already the source of truth for channel traffic on the Traffic Overview slide, and pulling a second, possibly-conflicting number for the same metric from the sheet would be redundant and confusing. Row 1 holds month labels (possibly spanning merged cells ŌĆö `forwardFillRow()` fills the blanks the Sheets values API returns for merged-cell columns after the first), row 2 is a `"Week 1"`/`"Week 2"`/etc sub-header (ignored), and data starts at row 3. `findMonthColumns()` matches row 1 against the target month by number, tolerant of name/abbreviation/numeric forms. DA/PA rows are found via `findRowByLabel()` and take the *first* non-empty value in the matched month's columns (not a sum), since they're point-in-time scores, not weekly counts; `null` (not `0`) if the row isn't found.
- `getSeoOverview(currentMonth, comparisonMonth, sheetId, email)` returns `{ keywordRankings: { current, previous, changes, hasData, error, comparisonError }, offPage: { current, previous, change, hasData, error }, domainAuthority, warnings }`. `routes/report.js`/`services/pptx.js` consume `keywordRankings`/`offPage` (`comparisonError` is read by `routes/report.js` only); `domainAuthority` is additive, not yet rendered by any slide. `warnings` is the flattened list of every human-readable warning generated across all sections (tab-not-found, label-not-found, date-column-not-found), available for logging. `hasData` is false both when the resolved tab has no date column for the **current** period and when no tab matched a category at all ŌĆö rendered as "Not configured" on the slide either way; only the warning text differs. `routes/report.js` skips calling this entirely (and synthesizes an all-`hasData: false`, no-`error` object) when the active client has no `sheetId` configured.
- In `services/pptx.js`, the SEO slide's keyword-bucket table renders `"No data"` (not `"0"`) for `previous` when `keywordRankings.previous === null`, and the summary paragraph switches to a current-period-only sentence in that case instead of dereferencing a null comparison object. The bucket label shown is `"Pending (100+)"`.
- Label/date-parsing/tab-resolution helpers are exported as `sheets.js#_internal` for unit testing only (`services/sheets.test.js`, run via `npm test` ŌåÆ `node --test`) ŌĆö not meant to be imported by application code.

### Environment / secrets

- `.env` holds `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `GOOGLE_REDIRECT_URI`, `PORT`, `SESSION_SECRET`, plus `GA4_PROPERTY_ID`/`GOOGLE_SHEET_ID`/`META_AD_ACCOUNT_ID`, the Meta app credentials `META_APP_ID`/`META_APP_SECRET`/`META_ACCESS_TOKEN`, the Google Ads credentials `GOOGLE_ADS_DEVELOPER_TOKEN`/`GOOGLE_ADS_LOGIN_CUSTOMER_ID`, and the Supabase credentials `SUPABASE_URL`/`SUPABASE_SERVICE_KEY` (service-role key, app-level ŌĆö used by `services/supabase.js`, see Supabase keyword data source above) ŌĆö gitignored. A missing `SUPABASE_URL`/`SUPABASE_SERVICE_KEY` isn't a boot failure; it just makes `getKeywordRankings` throw `SUPABASE_NOT_CONFIGURED`, which `routes/report.js` turns into a warning and the Supabase slide is simply omitted. `GA4_PROPERTY_ID`/`GOOGLE_SHEET_ID`/`META_AD_ACCOUNT_ID` are **not** read at request time anywhere in the app; they're used only by `services/clients.js#ensureSeeded()` to create a one-time "Default Client" in `clients.json` the very first time the app runs with no clients configured. `META_ACCESS_TOKEN` **is** read at request time, by `services/meta.js` ŌĆö it's app-level (one token for whoever runs the server), unlike GA4/Sheets which are per-signed-in-user OAuth. `GOOGLE_ADS_DEVELOPER_TOKEN`/`GOOGLE_ADS_LOGIN_CUSTOMER_ID` are also read at request time by `services/googleAds.js`, and are likewise app-level (one developer token/MCC ID for whoever runs the server) ŌĆö but unlike Meta, the actual API call authenticates with the signed-in user's own OAuth token (see Google Ads data layer above), not a shared app token. `META_APP_ID`/`META_APP_SECRET` aren't currently used by any code path (no OAuth flow for Meta yet) ŌĆö they're there for when token generation/refresh is automated. `SESSION_SECRET` signs the `express-session` cookie; it's auto-generated and appended to `.env` on first boot if absent (see OAuth / token handling above). None of these are in the `server.js` required-env check (only `GOOGLE_CLIENT_ID`/`GOOGLE_CLIENT_SECRET`/`GOOGLE_REDIRECT_URI` are required) ŌĆö a missing `META_ACCESS_TOKEN` just makes `getMetaAdsData` throw `META_NOT_CONFIGURED`, and a missing `GOOGLE_ADS_DEVELOPER_TOKEN`/`GOOGLE_ADS_LOGIN_CUSTOMER_ID` makes `getGoogleAdsData` throw `GOOGLE_ADS_NOT_CONFIGURED`, both of which `routes/report.js` turns into a warning, not a boot failure.
- `clients.json` (client profiles: `{ id, name, ga4PropertyId, sheetId, metaAdAccountId, googleAdsCustomerId, websiteUrl, createdAt }`) is gitignored, like `tokens.json` and `.env` ŌĆö it holds real GA4 property/sheet/Meta ad account/Google Ads customer IDs/website URL per client, shared across all signed-in users. Managed via `services/clients.js` / `routes/clients.js` (`GET/POST /api/clients`, `DELETE /api/clients/:id`); there's no update endpoint (delete + re-add to change a client's IDs). `metaAdAccountId` is stored `act_`-prefixed (see `normalizeMetaAdAccountId()` above) regardless of how the user typed it in; `googleAdsCustomerId` is stored with dashes stripped (see `normalizeGoogleAdsCustomerId()` above); `websiteUrl` is stored as-typed, no normalization.
- `tokens.json` (OAuth refresh/access tokens, keyed by signed-in user's email ŌĆö see OAuth / token handling above) is also gitignored and must never be logged; only `err.message`/`err.status`/`err.code` are logged for GA4, Sheets, Meta, and Google Ads API failures, never token or credential values.
- Meta access tokens generated from Graph API Explorer / a system user expire (short-lived tokens in hours, long-lived in ~60 days) and don't auto-refresh like the Google OAuth tokens do ŌĆö when `META_ACCESS_TOKEN` expires, every report for a client with a `metaAdAccountId` will show the Meta Ads slide warning until someone manually generates a new token and updates `.env` (the server does **not** need a restart to pick up the new value only if `.env` is reloaded ŌĆö in practice, restart `npm start` after editing `.env`). Google Ads doesn't have this problem for the OAuth side (it reuses the signed-in user's auto-refreshing token), but a new `GOOGLE_ADS_DEVELOPER_TOKEN` still requires a restart to pick up, same as any other `.env` change.
- A Google Ads developer token issued in **Test account** mode (the default until Google grants Basic Access) can only query test Google Ads accounts ŌĆö real customer IDs will fail with `GOOGLE_ADS_TOKEN_REQUIRED` until Basic Access is approved. This is expected and self-resolving; no code changes are needed once approval comes through (see the warning text in Google Ads data layer above).
- The Google Cloud OAuth client must be in "Testing" mode with the signing-in account added under **Test users**, and `http://localhost:3000/auth/callback` must be an authorized redirect URI, or sign-in fails with `403 access_denied`.
