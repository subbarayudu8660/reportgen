# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

**Maintenance instruction:** at the end of every session that changes this codebase (new routes, new data fields, changed slide layout/theme, changed auth/token behavior, etc.), update this file so it stays accurate — don't let it drift from the code.

## Commands

- `npm install` — install dependencies
- `npm run dev` / `npm start` — start the Express server (both run `node server.js`; no build step, no watcher)
- Server refuses to boot if `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, or `GOOGLE_REDIRECT_URI` are missing from `.env` (checked in `server.js`). `GA4_PROPERTY_ID`/`GOOGLE_SHEET_ID` are no longer required at boot — they're only used as one-time fallback defaults to seed the first client (see Client management below). If `SESSION_SECRET` is missing, `server.js` generates a random one and appends it to `.env` on first boot (see OAuth / token handling below) rather than refusing to start.
- There is no test suite, linter, or build tooling configured

## Architecture

ReportGen is a Node/Express app that authenticates against Google via OAuth 2.0, pulls GA4 report data for one or more clients, and generates a downloadable `.pptx` comparing two user-chosen reporting periods using `pptxgenjs`. Frontend is plain HTML/JS served statically from `public/`.

### Request flow

1. `public/app.js` polls `GET /auth/status` to decide whether to show the "Sign in" step or the "Choose Reporting Periods" step, and fetches `GET /api/clients` to populate the client selector independently of auth state.
2. `GET /auth/google` (routes/auth.js) redirects to Google's consent screen; `GET /auth/callback` exchanges the code for tokens via `services/googleAuth.js` and persists them to `tokens.json` at the repo root (gitignored, mode 0600).
3. `POST /api/generate-report` (routes/report.js) takes `{ clientId, currentMonth: "YYYY-MM", comparisonMonth: "YYYY-MM" }` — both periods are picked by the user and are not required to be adjacent months. It looks up the client via `services/clients.js#getClientById`, calls `services/ga4.js#getFullReportData(currentMonth, comparisonMonth, client.ga4PropertyId)` and (if `client.sheetId` is set) `services/sheets.js#getSeoOverview(currentMonth, comparisonMonth, client.sheetId)` in parallel, merges the SEO result onto the GA4 data as `data.seo`, pipes the combined object into `services/pptx.js#generateReportPptx`, and streams the resulting buffer back as a file download. There is no `GET /api/months` endpoint — the month/year selectors are built entirely client-side in `public/app.js`. Neither GA4 nor Sheets service reads `GA4_PROPERTY_ID`/`GOOGLE_SHEET_ID` from `.env` directly anymore — every call is parameterized by the active client's IDs, and there is no hardcoded "previous month" derivation anywhere in the backend.

### Client management (`services/clients.js`, `routes/clients.js`)

- Clients are persisted as a flat array in `clients.json` at the repo root (gitignored, like `tokens.json`) — `{ id, name, ga4PropertyId, sheetId, createdAt }`. `id` is a `crypto.randomUUID()`.
- `getClients()` calls `ensureSeeded()`, which creates a single "Default Client" from `.env`'s `GA4_PROPERTY_ID`/`GOOGLE_SHEET_ID` **only** the very first time the app runs (`clients.json` doesn't exist yet on disk). Once the file exists — even if the user later deletes every client down to an empty array — it is never reseeded, so `.env` values are a one-time bootstrap default, not a live fallback.
- `GET /api/clients` lists clients; `POST /api/clients` validates `name`+`ga4PropertyId` are present (`sheetId` is optional — a client with no sheet ID skips the Sheets call entirely in `routes/report.js` and renders the SEO slide as "Not configured", same as any other empty-data case) and appends a new client; `DELETE /api/clients/:id` removes one (404 if not found). There is no update/edit endpoint — editing a client means deleting and re-adding it.

### Period selection & report warnings (frontend)

- `public/index.html`/`public/app.js` show **two** independent Month+Year picker pairs, "Current Period" and "Comparison Period", instead of a single month dropdown. `populatePeriodSelectors()` builds both client-side: Current Period defaults to the most recently completed month (this month minus one, rolling back a year in January), and Comparison Period defaults to one month before that. Either period can be changed to any month/year in the current year or prior 2 years — they don't need to be adjacent. The Generate button starts `disabled` and `updateGenerateButtonState()` re-enables it once both period pairs have a value **and** a client is selected.
- The download filename is `WebrocketAI-Report-<CurrentMonth>-<CurrentYear>-vs-<ComparisonMonth>-<ComparisonYear>.pptx`, built from `monthLabel()` (services/pptx.js) server-side for `Content-Disposition` (routes/report.js) and mirrored client-side for the `<a download>` fallback name.
- `routes/report.js` computes a `warnings` array (plain user-facing sentences) after generating the pptx — one per: no GA4 data at all for the current period (`gaData.hasAnyGa4Data` false), no keyword ranking data (`seoData.keywordRankings.hasData` false), no off-page data (`seoData.offPage.hasData` false), and no comparison-period GA4 data (only added when the current period *does* have data, to avoid a redundant message alongside "no GA4 data at all"). It's sent as a `X-Report-Warnings` response header (JSON, `encodeURIComponent`-escaped, since header values must be ASCII) alongside the binary pptx — there's no separate JSON response for this, since the endpoint always returns the file on success. `public/app.js` reads and decodes this header, and renders each warning in the `#warnings` box (amber, non-blocking) alongside the download.

### OAuth / token handling (`services/googleAuth.js`)

- Tokens live in a single local JSON file (`tokens.json`), not a database, but the file holds **multiple users' tokens** keyed by Google account email: `{ "user@gmail.com": { access_token, refresh_token, expiry_date, scope, ... } }`. This app is still local-only by design (no external DB), but multiple people can sign in with different Google accounts against the same running server without clobbering each other's credentials.
- Every function that reads or writes tokens takes an `email` argument: `loadTokens(email)`, `saveTokens(email, tokens)`, `isAuthenticated(email)`, `hasSheetsScope(email)`, `getAuthorizedClient(email)`. There is no more "the" token — always pass the email of the user whose credentials you want.
- The signed-in user's email is identified via the Google userinfo endpoint (`GET https://www.googleapis.com/oauth2/v2/userinfo`, called with the freshly-exchanged access token) inside `handleCallback(code)`, which now returns `{ tokens, email }` instead of just `tokens`. `SCOPES` includes `openid` and `.../auth/userinfo.email` (in addition to `analytics.readonly` and `spreadsheets.readonly`) so that call succeeds.
- Google only returns a `refresh_token` on the *first* consent grant for a given account; `handleCallback` and the `client.on('tokens', ...)` listener in `getAuthorizedClient` both special-case this by merging new tokens onto that email's existing stored `refresh_token` rather than overwriting it.
- `getAuthorizedClient(email)` throws `NOT_AUTHENTICATED` (no email, or no stored refresh token for that email) or `AUTH_EXPIRED` (refresh failed) — callers key off `err.code` to return the right HTTP status.
- `hasSheetsScope(email)` checks that email's stored token `scope` string for the Sheets scope. Tokens saved before the Sheets scope was added won't have it; `GET /auth/status` reports `needsReauth: true` in that case (authenticated but missing scope), and `public/app.js` shows the sign-in step with a prompt instead of the generate step. `services/sheets.js` also guards each call with its own `ensureSheetsScope(email)`, throwing `SHEETS_SCOPE_MISSING` (mapped to HTTP 401 in `routes/report.js`) as a second line of defense.
- **Which user is "current" is a session concern, not a token-store concern.** `server.js` wires up `express-session` (cookie name `reportgen.sid`, backed by `SESSION_SECRET` from `.env`) before the static/route middleware. On first boot, if `SESSION_SECRET` is absent, `server.js` generates one with `crypto.randomBytes(32)` and appends it to `.env` — it does not block startup or require the user to set it manually. `routes/auth.js#GET /callback` stores the resolved email as `req.session.email` after a successful OAuth exchange; every downstream route that needs Google API access reads `req.session.email` and threads it down as the `email` argument (`routes/report.js` → `services/ga4.js#getFullReportData(..., email)` / `services/sheets.js#getSeoOverview(..., email)` → every individual GA4/Sheets fetch function → `getAuthorizedClient(email)`). `GET /auth/status` reads `req.session.email` and returns `{ authenticated, needsReauth, email }` for the current session only — two different browser sessions (e.g. two users, or one user in a normal tab vs. incognito) see independent auth state even though they share the same `tokens.json`.
- `GET /auth/logout` calls `req.session.destroy()`, clears the `reportgen.sid` cookie, and redirects to `/` — this only ends the *current session*, it does not delete anything from `tokens.json`, so other signed-in users (or the same user in another browser/session) are unaffected. It's a plain `<a href="/auth/logout">` (not a fetch call) so the redirect does a full page reload, which re-runs `checkAuthStatus()` and lands back on the sign-in screen. There is no endpoint to remove a user's stored tokens entirely; the only way to fully revoke is to edit `tokens.json` or revoke access from the Google Account permissions page.
- `public/index.html` has a `#user-bar` fixed to the top-right corner of the page (email + "Sign Out" link, styled `.btn-outline` — white/grey/dark, deliberately unobtrusive), outside `.container` so it stays in place independent of card layout/scroll. `checkAuthStatus()` in `public/app.js` toggles it on `authenticated` from `GET /auth/status` (shown even when `needsReauth` is true, since signing out is a valid way to recover from that state) and fills in the email from the same response.
- Migration note: `tokens.json` files written before this change are a flat single-token object (no email keys), which `loadAllTokens()` silently treats as "no known users" — anyone who was signed in under the old scheme must sign in again once after this change ships.

### GA4 data layer (`services/ga4.js`)

- Calls the GA4 Data API REST endpoint directly (`analyticsdata.googleapis.com/v1beta/properties/{id}:runReport`) via `fetch` with a bearer token from the OAuth client, rather than using the `@google-analytics/data` gRPC SDK — simpler to reason about with user-OAuth (vs. service-account) credentials. `runReport(body, propertyId)` takes the property ID as an explicit argument (the active client's `ga4PropertyId`) rather than reading `GA4_PROPERTY_ID` from `.env`.
- `runReport()` throws `GA4_API_ERROR` on any non-2xx response, with `.status` (HTTP status) and `.apiErrorStatus` (the GA4 API's own error status string parsed from the JSON body, e.g. `PERMISSION_DENIED`, `NOT_FOUND`, `INVALID_ARGUMENT`, `RESOURCE_EXHAUSTED`, `UNAUTHENTICATED` — `null` if the body wasn't JSON). `mapGa4Error(err)` maps that error to a specific, user-actionable `{ httpStatus, message }` pair (403 permission/property-access, 404 property not found, 400 invalid property ID format, 429 quota exceeded, 401 expired session, else a generic "unexpected error (code)" at 502); `routes/report.js` logs the raw `status`/`apiErrorStatus`/`message` server-side and returns `mapGa4Error`'s result as the JSON error response. Any new GA4-error-handling case should extend `mapGa4Error`, not add ad hoc status checks in `routes/report.js`.
- All four report sections are fetched in parallel via `Promise.all` inside `getFullReportData(currentMonth, comparisonMonth, propertyId)`: organic search metrics, ecommerce funnel, top landing pages, and channel traffic overview — each fetched for both the current and comparison month (both user-supplied; there is no assumption that the comparison month is adjacent to the current one) so percent-change can be computed.
- `getTopLandingPages` does a second query (`getSessionsByPagePath`) scoped to the current month's top-10 page paths to get comparison-month sessions for the *same* pages (not the comparison month's own top 10) — pages with no comparison-month traffic get `prevSessions: null` / `change: null`, rendered as "N/A".
- Ecommerce `hasData` is computed by summing all funnel metrics + revenue and checking `> 0`, not by row count — GA4 can return rows for zero-count events, so row presence alone is not a reliable "no ecommerce" signal.
- `getFullReportData` also fetches an unfiltered total-sessions count for both months (`getMonthSessionTotal`) to derive `hasAnyGa4Data`/`comparisonMonthHasGa4Data` — these are channel/event-agnostic, unlike the per-section `hasData` flags, so they answer "did GA4 track *anything* this month" (e.g. before the property existed) rather than "did this specific section have rows". When `comparisonMonthHasGa4Data` is false, `organicSearch.changes`, `ecommerce.changes`, and each `trafficOverview` row's `change` are forced to `null` instead of being computed against a false "0" baseline — a real zero-traffic comparison month and a not-yet-tracked comparison month must not both render as "+100%"/"0%". The returned object's top-level fields are `currentMonth`/`comparisonMonth` (not `month`/`previousMonth`).

### PPTX generation (`services/pptx.js`)

- Slide order is fixed: Traffic Overview → SEO Performance Overview → Organic Search Performance → Ecommerce Funnel Performance → Top Landing Pages, built by `generateReportPptx(data)` from the shape returned by `getFullReportData` plus `data.seo` (see `getSeoOverview`). The SEO slide places two tables side by side (keyword ranking buckets on the left, off-page submissions on the right) via `addTable`'s `opts.x/w`.
- Theme is a light "agency deliverable" style, not a dark dashboard: `#F8F9FA` slide background, indigo `#4361EE` accent (titles, title underline, table header fill), dark grey `#212529` body text, alternating white/`#F1F3F5` table rows, and a small grey footer (`WebrocketAI — [Month Year] Report`) bottom-right on every slide. Shared chrome comes from `addSlideChrome`/`addSummary`/`addTable`/`addFooter` — new slides should reuse these rather than styling ad hoc. `monthLabel()` and `addFooter()` are keyed off `data.currentMonth` specifically.
- % change cells are built with `pctCell(n)`, which returns a `{ text, highlight }` object (bold, green `#2DC653` / red `#E63946` / grey `#6C757D` for null-or-NaN) rather than a plain string; `addTable` special-cases these object cells vs. plain string cells when rendering. Any new % change column should go through `pctCell`, not a hand-formatted string, to keep the color-coding consistent.
- Every slide function checks the relevant `hasData` flag from the GA4 layer and renders a placeholder (not a blank/error slide) when a data source is empty — e.g. the ecommerce slide shows a "Not configured" table if no ecommerce events exist. When adding new data sections, follow this same has-data branch pattern instead of assuming data is present.
- Separately from per-section `hasData`, the Traffic Overview, Organic Search, and Ecommerce slides also check `data.comparisonMonthHasGa4Data`: when false, the "Comparison Period" table column reads `"No comparison data"` instead of a formatted zero, the summary paragraph swaps to a current-period-only sentence plus an explicit "No comparison period data available" note, and `pctCell` receives `null` (already forced upstream in `ga4.js`) so the % column renders "N/A" rather than a misleading percentage. The Top Landing Pages slide appends the same note to its summary when this flag is false (per-page current/comparison comparison there already uses its own per-page `prevSessions: null` handling from `ga4.js`).
- Summary paragraphs are string-built from the actual computed numbers (not templated GA4 responses) — keep this dynamic-summary approach when extending, e.g. singling out the top-growth or top-decline row. Wording is intentionally period-agnostic ("comparison period", not "previous/prior month") since the comparison period is user-chosen and may not be adjacent to the current one.
- `pagePath === '/'` is relabeled `Homepage` for display only (`displayPagePath`); raw `pagePath` values are still used for GA4 filtering upstream in `ga4.js`.
- `addTable` takes an optional 4th `opts` arg (`{ x, y, w }`, defaulting to the original full-width position) so slides can place two tables side by side instead of one full-width table per slide.

### Google Sheets data layer (`services/sheets.js`)

- Reads two tabs of the spreadsheet at a given `sheetId` (the active client's `sheetId`, not read from `.env`) via the Sheets API v4 REST endpoint (`sheets.googleapis.com/v4/spreadsheets/{id}/values/{range}`) using the same bearer token pattern as `ga4.js`, not the `googleapis` SDK. `getSheetValues(range, sheetId)` throws if `sheetId` is falsy.
- `'Keyword Ranking Report'!A10:ZZ2000`: row 10 is the header with dates as `M/D/YYYY` strings across columns (parsed manually in `parseSheetDate`, not via `Date()`, to avoid locale ambiguity); keywords start at row 11 in column A. For a given month, `latestColForMonth` picks the *most recent* date column that falls in that month (sheets can have multiple snapshot dates per month) — not the first or an average.
- Each keyword's ranking value is bucketed by `classifyRank`: Top 10 / 11-30 / 31-50 / 51-100 / Pending, where blank, non-numeric, and any value containing `"100+"` are all treated as Pending.
- `'KPI_2'!A1:ZZ500`: row 1 is dates per-day across columns, rows 2+ are named activity types (Off Page Submissions, SBMs, Directory Submissions, etc.). All activity rows are summed across every date column that falls in the target month — this is a total across activity types, not per-type.
- `getSeoOverview(currentMonth, comparisonMonth, sheetId)` returns `{ keywordRankings: { current, previous, changes, hasData }, offPage: { current, previous, change, hasData } }`. `hasData` is false when the sheet has no date column in that month (e.g. months before the sheet's tracking started) — rendered as "Not configured" on the slide, same pattern as the GA4 ecommerce slide. `routes/report.js` skips calling this entirely (and synthesizes an all-`hasData: false` object) when the active client has no `sheetId` configured, since a `sheetId` is optional per client.

### Environment / secrets

- `.env` holds `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `GOOGLE_REDIRECT_URI`, `PORT`, `SESSION_SECRET`, plus `GA4_PROPERTY_ID`/`GOOGLE_SHEET_ID` — gitignored. `GA4_PROPERTY_ID`/`GOOGLE_SHEET_ID` are **not** read at request time anywhere in the app; they're used only by `services/clients.js#ensureSeeded()` to create a one-time "Default Client" in `clients.json` the very first time the app runs with no clients configured. `SESSION_SECRET` signs the `express-session` cookie; it's auto-generated and appended to `.env` on first boot if absent (see OAuth / token handling above). None of these four are in the `server.js` required-env check (only `GOOGLE_CLIENT_ID`/`GOOGLE_CLIENT_SECRET`/`GOOGLE_REDIRECT_URI` are required).
- `clients.json` (client profiles: `{ id, name, ga4PropertyId, sheetId, createdAt }`) is gitignored, like `tokens.json` and `.env` — it holds real GA4 property/sheet IDs per client, shared across all signed-in users. Managed via `services/clients.js` / `routes/clients.js` (`GET/POST /api/clients`, `DELETE /api/clients/:id`); there's no update endpoint (delete + re-add to change a client's IDs).
- `tokens.json` (OAuth refresh/access tokens, keyed by signed-in user's email — see OAuth / token handling above) is also gitignored and must never be logged; only `err.message`/`err.status` are logged for GA4 and Sheets API failures, never token or credential values.
- The Google Cloud OAuth client must be in "Testing" mode with the signing-in account added under **Test users**, and `http://localhost:3000/auth/callback` must be an authorized redirect URI, or sign-in fails with `403 access_denied`.
